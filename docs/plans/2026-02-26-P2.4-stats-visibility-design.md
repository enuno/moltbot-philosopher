# P2.4 Design: Daily Stats Visibility & Production Readiness

**Version:** 1.0
**Date:** 2026-02-26
**Phase:** P2.4 (15h estimate)
**Status:** Design Approved (Ready for Implementation)

## Overview

P2.4 is Phase 2's final graduation phase. It expands the `/stats` endpoint into a production-grade metrics surface that provides trend analysis, per-agent engagement velocity metrics, and a dashboard-friendly JSON format. Upon completion, the engagement pipeline graduates to production readiness.

**Core principle:** Observable, robust, performant — all three. No metrics hacks. Schema locked.

### What "Production Graduated" Means

P2.4 satisfies four dimensions of production readiness:

1. **Operational**: Runs 24/7 with basic health signals (uptime, staleness, version)

2. **Robust**: Handles missing data, corrupted state, and upstream failures without breaking or emitting garbage

3. **Performant**: All aggregations are O(1) or O(n_agents) — no scaling with thread volume

4. **Documented & Validated**: End-to-end tests confirm the entire chain works; schema is locked and stable

---

## Goals

- Expand `/stats` to expose trend analysis (top topics, top threads, engagement velocity)

- Add per-agent engagement velocity metrics (`comments_per_hour`, `threads_touched`, `threads_started`)

- Lock a stable, dashboard-friendly JSON schema for `/stats`

- Ensure resilience: explicit staleness flags, empty arrays on missing data, no NaN/null responses

- Validate the entire engagement pipeline with E2E tests (P2.1 → P2.2 → P2.3 → /stats)

## Architecture

### Layer 1: State Management (`state-manager.ts`)

**Responsibility:** Maintain rolling 24h aggregates from engagement events.

**What it tracks:**

- Per-topic thread counts, quality scores, sentiment, engagement velocity

- Per-agent metrics: `comments_per_hour`, `threads_touched`, `threads_started`

- Per-agent topic affinities (topic IDs and quality scores)

- Quality distribution histogram (3 buckets: 0–0.4, 0.4–0.7, 0.7–1.0)

- Sentiment statistics on controversial threads (mean, variance, sample count)

- Service timestamps: `last_heartbeat_at`, `last_metrics_rollup_at`

**When it updates:**

- Rollup job runs during each heartbeat (no separate cron)

- Maintains a 24h sliding window (older data pruned)

- Handles missing/corrupted state gracefully (returns empty/zero fallbacks)

**Data structure:**

```typescript
interface RollingMetrics {
  lastHeartbeatAt: Date;
  lastMetricsRollupAt: Date;
  topicCounts: Map<string, number>;                // topic_id → thread count
  topicQuality: Map<string, { sum: number; count: number }>;
  topicSentiment: Map<string, { sum: number; count: number }>;
  topicVelocity: Map<string, number>;              // topic_id → engagement velocity
  agentVelocity: Map<string, AgentVelocityMetrics>;
  agentTopics: Map<string, Map<string, TopicMetrics>>;
  qualityDistribution: [number, number, number];    // buckets: 0–0.4, 0.4–0.7, 0.7–1.0
  controversialSentiment: { sum: number; count: number; variance: number };
}

interface AgentVelocityMetrics {
  commentsPerHour: number;
  threadsTouched: number;
  threadsStarted: number;
  avgQualityScore: number;
  controversialThreadCount: number;
}

interface TopicMetrics {
  threadCount: number;
  avgQualityScore: number;
}

```

### Layer 2: Metrics Builder (`stats-builder.ts`)

**Responsibility:** Compute derived metrics from state (cheap, in-memory operations).

**Functions:**

- `buildTopicTrends(metrics)` → top N topics sorted by velocity

- `buildThreadTrends(metrics, threadIndex)` → top N threads by quality & engagement

- `buildAgentMetrics(metrics)` → per-agent summaries

- `buildQualityDistribution(metrics)` → histogram + sentiment stats

**Key property:** All operations assume state is already aggregated. No DB queries, no nested loops over threads.

### Layer 3: Stats Endpoint (`stats-endpoint.ts`)

**Responsibility:** Serve the `/stats` JSON response.

**Behavior:**

- GET `/stats` → returns `StatsResponse` (see schema below)

- Calls `stats-builder` to derive metrics from state

- Includes staleness check: sets `data_freshness.is_stale = true` if metrics older than 5 minutes

- Returns empty arrays + `is_stale: true` if state is missing or corrupted

- Never throws 500; gracefully degrades to partial data

## JSON Schema (Locked)

### Top-level Response

```typescript
export interface StatsResponse {
  service: StatsServiceInfo;
  summary: StatsSummary;
  trends: StatsTrends;
  agents: StatsAgentsSection;
  quality: StatsQualitySection;
}

```

### Section 1: Service Info

```typescript
export interface StatsServiceInfo {
  name: string;                              // "engagement-service"
  version: string;                           // "2.0.0"
  timestamp: string;                         // ISO 8601
  uptime_seconds: number;
  data_freshness: {
    last_heartbeat_at: string;              // ISO 8601
    last_metrics_rollup_at: string;         // ISO 8601
    is_stale: boolean;                      // true if > 5 min old
  };
}

```

### Section 2: Summary

```typescript
export interface StatsSummary {
  window_hours: number;                      // e.g., 24
  total_posts_seen: number;
  total_threads_tracked: number;
  total_comments_seen: number;
  engagement_events: {
    replies_posted: number;
    threads_continued: number;
    proactive_posts_published: number;
  };
  average_quality_score: number;             // 0–1
  controversial_thread_count: number;
}

```

### Section 3: Trends

```typescript
export interface StatsTrends {
  top_topics: StatsTopicTrend[];            // top 5–10 topics by velocity
  top_threads: StatsThreadTrend[];          // top 5–10 threads by quality
}

export interface StatsTopicTrend {
  topic_id: string;                         // "ai_safety"
  label: string;                            // "AI Safety and Alignment"
  thread_count: number;
  avg_quality_score: number;                // 0–1
  avg_sentiment: number;                    // -1 to 1
  engagement_velocity: number;              // comments/hour
}

export interface StatsThreadTrend {
  thread_id: string;
  root_post_id: string;
  quality_score: number;                    // 0–1
  is_controversial: boolean;
  controversial_tags: string[];             // topic IDs
  total_comments: number;
  unique_participants: number;
  engagement_velocity: number;              // comments/hour
}

```

### Section 4: Per-Agent Metrics

```typescript
export interface StatsAgentsSection {
  window_hours: number;
  per_agent: StatsAgentMetrics[];
}

export interface StatsAgentMetrics {
  agent_type: string;                       // "classical", "cyberpunk", etc.
  engagement_velocity: {
    comments_per_hour: number;
    threads_touched: number;
    threads_started: number;
  };
  quality: {
    avg_quality_score: number;              // 0–1
    controversial_thread_count: number;
  };
  topics: StatsAgentTopicSummary[];
}

export interface StatsAgentTopicSummary {
  topic_id: string;
  thread_count: number;
  avg_quality_score: number;                // 0–1
}

```

### Section 5: Quality Distribution & Sentiment

```typescript
export interface StatsQualitySection {
  window_hours: number;
  distribution: {
    bucket_0_0_4: number;                   // count of threads with score [0, 0.4)
    bucket_0_4_0_7: number;                 // count of threads with score [0.4, 0.7)
    bucket_0_7_1_0: number;                 // count of threads with score [0.7, 1.0]
  };
  sentiment_on_controversial: {
    avg_sentiment: number;                  // -1 to 1
    variance: number;
    sampled_threads: number;
  };
}

```

## Robustness & Error Handling

### Graceful Degradation

- If state is missing or corrupt:
  - `data_freshness.is_stale = true`

  - All arrays return empty: `top_topics: []`, `top_threads: []`, `per_agent: []`

  - Numeric fields return 0 or sensible defaults (not NaN, null, or undefined)

  - HTTP 200 always returned (never 500 for missing data)

### Staleness Definition

- Staleness threshold: `now - last_metrics_rollup_at > 5 * HEARTBEAT_INTERVAL`

- Default heartbeat interval: 30 seconds → 5-minute staleness threshold

- If `is_stale = true`, consumers know to refresh soon or use cached data with caution

### Edge Cases

- **Empty thread history**: `total_threads_tracked: 0`, arrays empty, quality distribution all zeros

- **No controversial threads**: `avg_sentiment: 0`, `variance: 0`, `sampled_threads: 0`

- **Agent with no engagement**: `comments_per_hour: 0`, `threads_touched: 0`, `topics: []`

---

## Performance Requirements

- **Aggregation logic**: O(1) per heartbeat update (append to rolling window, update running sums)

- **Endpoint latency**: < 50ms for /stats (in-memory reads only)

- **State size**: < 10 MB even with 1000s of threads (maps + aggregates, no full thread storage)

- **Memory per agent**: ~1 KB baseline + topic affiliations (8 topics max)

## Testing & Validation

### End-to-End Test Coverage

1. **Engagement pipeline**: Post created → relevance scored (P2.1) → quality metrics updated (P2.2) → proactive posting evaluated (P2.3) → /stats reflects new data

2. **Staleness tracking**: Advance time in test, verify `is_stale` flag flips

3. **Robustness**: Delete state, call /stats, verify graceful response with empty data

4. **Per-agent velocity**: Simulate multiple comments per agent, verify `comments_per_hour` correct

5. **Topic trends**: Create threads on multiple topics, verify ranking by velocity

6. **Quality distribution**: Create threads with various quality scores, verify histogram buckets correct

- Mock HTML dashboard consumes `/stats` JSON

- Renders top topics, per-agent metrics, quality histogram

- Verify no rendering errors, all fields present and formatted
